Key User Adaptation: Enable Your App

Here's what you have to consider when developing apps that support key user adaptation.

Key user adaptation is a feature of SAPUI5 flexibility that allows end users to personalize object pages when running an app in the SAP Fiori launchpad and key users without technical knowledge to easily make UI changes for all users of an app. For more information, see SAPUI5 Flexibility: Adapting UIs Made Easy.

Note

If your app is based on SAP Fiori elements, it automatically supports key user adaptation. You only have to consider the following if you use extension points to modify it, for example to add an additional view. You can adapt the UI of apps based on SAP Fiori elements using the SAPUI5 Visual Editor in SAP Web IDE. For more information, see SAPUI5 Visual Editor in the SAP Web IDE Full-Stack guide.

Tip

When viewing a sample in the SAPUI5 Demo Kit, you can check whether a control is enabled for key user adaptation by choosing the wrench symbol (Adapt UI) at the top right. If UI elements of the sample get highlighted when you hover over or select them, it's an indicator that the control is enabled for key user adaptation. (You can also try out the UI adaptation features directly in the sample.)

Configure the variant management control in your app to enable saved views (aka "control variants")

If users should be able to save their UI changes as different views (control variants), you have to configure the sap.ui.fl.variants.VariantManagement control.

All you have to do is add this control to an appropriate location in your app, and assign the desired UI container as a target in a for association.

Very important: Use stable IDs

As stable IDs are used to identify the controls that a key user can adapt using key user adaptation, you have to use stable IDs for each control used in the supported controls and in each view of your app.

We recommend that you also use stable IDs for all other controls and views of your app, to ensure that they can be adapted in future using key user adaptation. For more information, see Stable IDs: All You Need to Know.

Caution

Never change the IDs when creating the next version of your app if it was already delivered. Otherwise you'd risk inconsistencies and the loss of UI changes that a key user might have done to the delivered version.

As of SAPUI5 version 1.58: Use the flexEnabled flag

As of app descriptor schema version 1.12 (SAPUI5 version 1.58), the flexEnabled flag in the sap.ui5 section of the manifest.json indicates if the application supports key user adaptation and developer adaptation. If you set this flag, SAP Web IDE is able to validate your IDs. If it finds IDs that are not stable, you can either provide stable ID manually, or SAP Web IDE can generate stable IDs for you. For more information, see Validation of Stable IDs in the SAP Web IDE Full-Stack guide.

Caution

Do not change the flexEnabled flag to false after your application has already been delivered. This would lead to incompatibilities and regressions, if a key user has adapted the UI in the meantine.

Should the key user be able to add additional UI elements?

To enable the key user to make visible additional properties of an OData entity in key user adaptation (typically when the key user adds fields or custom fields), you have to do the following:

  • Use data binding in the container where you want to enable this.

  • Make sure that the related OData model is the default model of your app component. For more information, see Assigning the Model to the UI.

Use asynchronous loading of views to improve performance

To enable processing of UI changes directly on XML views, use the asynchronous loading of views. For more information, see Instantiating Views.

Tip

Using the rule Asynchronous XML views in the Support Assistant, you can check whether asynchronous loading is used for all views. For more information, see Support Assistant.

Restrict adaptation for certain controls

It can be a good idea to exclude certain controls from key user adaptation. Examples:
  • Controls for standard actions that should not be removed (for example, the Close button)
  • Controls that should not be changed, because their properties are changed dynamically by the application coding, which will consequently overwrite key user changes
To achieve this, you need instance-specific design time metadata. You attach these metadata via the sap.ui.dt namespace and the designtime attribute to the xml node of the control you want to restrict, or via CustomData. There are two types of restrictions:
  • Control cannot be removed (recommended, for example, for standard actions):

    Required metadata: not-removable

  • Control cannot be changed (recommended, for example, if the control properties are changed dynamically by the application coding):

    Required metadata: not-adaptable

Note

Controls that have already been delivered with previous app versions should not be switched to not-removable or not-adaptable later. Reason: Setting these metadata later will not affect existing changes and might even cause regressions.

The process to define the design time metadata depends on your scenario:

XML View

In this case, you need to specify metadata via the sap.ui.dt namespace. Example: 

<core:View
	...
	xmlns:sap.ui.dt="sap.ui.dt"
>
	...
	<SomeControl sap.ui.dt:designtime="not-removable" />
	<SomeOtherControl sap.ui.dt:designtime="not-adaptable" />
	...
</core:View>

Control Is Instantiated by JavaScript Code

In this case, you need to provide the instance-specific design time metadata as custom data. Example:

new SomeControl({
	//other settings
	customData :  [new CustomData({
		key : "sap-ui-custom-settings",
		value : {
			"sap.ui.dt" : {
				"designtime": "not-removable" || "not-adaptable"
			}
		}
	})];
});

Something Isn’t Working Like It Should?

Please check the troubleshooting information. For more information, see Troubleshooting.

Note

Bear in mind that depending on the controls used, it might be necessary that key users make their UI changes in edit mode as well as in display mode of the app. Exception: smart forms show changes made in one mode automatically in the other.

For more information about how key user adaptation works for key users and how it gets enabled in the SAP Fiori launchpad, search for Adapting SAP Fiori UIs at Runtime and Enabling UI Adaptation at Runtime in the documentation for your SAP NetWeaver version on the SAP Help Portal.
